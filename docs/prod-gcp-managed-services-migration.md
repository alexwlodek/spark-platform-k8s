# PROD GCP Managed Services Migration

This document tracks the PROD migration from DEV-style in-cluster infrastructure to GCP-managed services while keeping one repo, one GitOps flow, and the current DEV experience intact.

## Phase 1 - Analysis

### Findings

- DEV is the functional reference architecture for the data path: Kafka -> Spark -> object storage -> Nessie -> Trino -> Metabase.
- PROD already has GKE, Argo CD, External Secrets, and GCP Secret Manager integration, but the data platform layer was still modeled around AWS/S3/RDS placeholders.
- The repo already separates environments cleanly through `clusters/<env>` and `values/<env>`, so the migration can stay values-driven without template `if/else` branches.
- The most important stateful DEV components to replace in PROD are MinIO and PostgreSQL for Nessie metadata.
- Trino and the Spark streaming job were still carrying S3-shaped configuration in shared values, which made PROD look cloud-native in name only.
- PROD did not deploy `storage-nessie` or `bi-trino` yet, so those had to be wired into the prod app-of-apps set.

### Migration map

| Current DEV component | Purpose | PROD GCP equivalent | Impact | Required changes | Transparent? |
| --- | --- | --- | --- | --- | --- |
| `storage-minio` | Iceberg warehouse and Spark checkpoints | Google Cloud Storage bucket | Medium | Terraform bucket, Spark GCS connector, Trino native GCS config, Workload Identity | No, config adaptation required |
| `storage-nessie-db` | PostgreSQL metadata backend for Nessie | Cloud SQL for PostgreSQL | Medium | Terraform Cloud SQL, workload identity, Cloud SQL proxy sidecar, secret flow | Mostly transparent to Nessie app |
| `platform-minio-creds` secret flow | DEV-only S3 credentials | Workload Identity for GCS | Medium | Remove PROD MinIO secret dependency, annotate runtime KSAs | No, credential model changes |
| `platform-nessie-db-creds` secret flow | DEV-only in-cluster Postgres bootstrap creds | Cloud SQL user generated by Terraform + Secret Manager | Low | Remove unused PROD ExternalSecret, keep DEV secret contract | Yes for DEV, no for PROD plumbing |
| `bi-trino` S3 catalog config | Query Iceberg through Nessie | Trino native GCS filesystem | Medium | Prod overlay uses `fs.native-gcs.enabled` and ADC | No, catalog config changes |
| `streaming-pipeline` S3A + Iceberg AWS config | Spark checkpoints and Iceberg writes | GCS connector + Iceberg `GCSFileIO` | Medium | Prod overlay switches to `gs://`, GCS connector, ADC | No, Spark config changes |
| `streaming-kafka` | DEV local broker | Managed Kafka endpoint on GCP or external service | High | External endpoint input, optional Terraform later | Not implemented in this change |
| `bi-metabase` with PVC-backed H2 | BI UI state | Prefer Cloud SQL-backed app DB if promoted to PROD | Medium | Chart needs external DB support | Not implemented in this change |
| `logging-elasticsearch` | Self-hosted log store | Prefer managed Elastic/OpenSearch or Cloud Logging for real PROD | High | Bigger observability redesign | Deferred |
| `monitoring` PVC-backed Prometheus/Grafana | Metrics retention and dashboards | Managed Service for Prometheus / hosted Grafana optional | High | Observability redesign | Deferred |

## Phase 2 - Architecture Proposal

### Variant A

- Keep S3-shaped app configuration in PROD.
- Use GCS interoperability / HMAC-style credentials for Spark and Trino.
- Connect Nessie directly to Cloud SQL private IP.

Pros:

- Smaller app-level change surface.
- Easier short-term lift from MinIO.

Cons:

- Keeps AWS-oriented config in a GCP platform.
- More secret material to manage.
- Worse long-term fit for Iceberg and Trino on GCP.

### Variant B

- Use native GCS access for Spark and Trino.
- Use GKE Workload Identity instead of static object-store credentials.
- Run Nessie in-cluster, but move its metadata database to Cloud SQL and connect through the Cloud SQL Auth Proxy sidecar.

Pros:

- Best GCP fit.
- Strongest reduction in credential sprawl.
- Keeps DEV intact while making PROD cloud-native through overlays only.

Cons:

- Requires moderate value/chart refactoring.
- Spark and Trino need explicit GCS configuration.

### Recommendation

Choose Variant B.

Reasoning:

- Simplicity: once Workload Identity is in place, Spark and Trino stop carrying object-store secrets.
- Production readiness: Cloud SQL + GCS + Workload Identity is a durable GKE pattern.
- Consistency with DEV: the logical architecture stays the same even though the backing services change.
- Maintainability: storage backend differences live in `values/dev` vs `values/prod`, not in env-specific templates.

## Phase 3 - Infrastructure Foundation

### Findings

- The repo already had split Terraform stacks for `network` and `gke`.
- Private Service Access for Cloud SQL was not configured yet.
- No Terraform stack existed for managed platform data services.

### Proposed changes

- Extend `network` with Private Service Access plumbing for Cloud SQL private IP.
- Add `infra/envs/gcp/platform` for:
  - the production lake bucket in GCS
  - the Nessie Cloud SQL instance, database, and user
  - runtime GSAs for Spark/Trino and Nessie
  - Workload Identity bindings
  - Secret Manager secret material for Nessie DB credentials
- Extend the production helper script so `--infra` now means `network -> gke -> platform`.

## Phase 5 - Application Compatibility

### Findings

- Shared values contained MinIO/S3-specific defaults that leaked into PROD.
- Spark and Trino only needed config changes, not application code changes.
- The biggest compatibility issue was backend configuration shape, not business logic.

### Proposed changes

- Keep app code shared.
- Move backend-specific Spark and Trino storage config into environment overlays.
- DEV keeps:
  - MinIO
  - S3 credentials via Kubernetes Secrets
  - in-cluster PostgreSQL for Nessie metadata
- PROD uses:
  - GCS
  - Workload Identity
  - Cloud SQL for Nessie metadata

- The PROD streaming values are now GCS-ready, but the `streaming-pipeline` app is intentionally not added to the prod root yet because a real managed Kafka endpoint is still missing from the repo.
- This keeps PROD reviewable and avoids deploying a knowingly broken Spark workload.

## Short Migration Checklist

1. Copy `terraform.tfvars.example` into:
   - `infra/envs/gcp/network/terraform.tfvars`
   - `infra/envs/gcp/gke/terraform.tfvars`
   - `infra/envs/gcp/platform/terraform.tfvars`
2. Apply Terraform in order:
   - `network`
   - `gke`
   - `platform`
3. Seed or confirm the remaining non-generated Secret Manager payloads:
   - `spark-platform-prod-argocd`
   - `spark-platform-prod-cert-manager-cloudflare`
   - `spark-platform-prod-monitoring-grafana`
4. Run `scripts/prod-deploy.sh --bootstrap` or `scripts/prod-deploy.sh --all`.
5. Verify Argo CD apps:
   - `storage-nessie`
   - `bi-trino`
   - `security-external-secrets`
   - `security-gcp-secretmanager`
6. Verify Terraform outputs:
   - lake bucket name
   - Cloud SQL connection name
   - runtime GSA emails
7. Only after a managed Kafka endpoint is ready, add `streaming-pipeline` to the PROD root and enable the Spark workload.
